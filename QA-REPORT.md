## 2. ⚠️ НАЙДЕННЫЕ ПРОБЛЕМЫ (Critical & Major)

### 🔴 CRITICAL: Пароли в открытом виде на клиенте (main.js)
```javascript
var accounts = {
    'admin@numino.ru':  { pass: 'admin123', ... },
    'owner@numino.ru':  { pass: 'owner123', ... },
};
```
- **Severity**: CRITICAL
- **Описание**: Учётные данные (email + пароль) хранятся **в открытом виде** в клиентском JavaScript-файле. Любой посетитель сайта может открыть инструменты разработчика и получить доступ к учётным данным.
- **Риск**: Несанкционированный доступ к панели управления автопарком.
- **Рекомендация**: Использовать Firebase Authentication с серверной верификацией; никогда не хранить пароли на клиенте.

### 🔴 CRITICAL: Форма обратной связи не отправляет данные
```javascript
contactForm.addEventListener('submit', function (e) {
    e.preventDefault();
    // Просто показывает "Отправлено!" и сбрасывает форму
    btn.textContent = 'Отправлено!';
    setTimeout(function () { contactForm.reset(); }, 2200);
});
```
- **Severity**: CRITICAL
- **Описание**: Форма обратной связи имитирует отправку, но **фактически ничего никуда не отправляется**. Сообщения пользователей теряются.
- **Риск**: Потеря потенциальных клиентов, вводящий в заблуждение интерфейс.
- **Рекомендация**: Реализовать отправку через Email-сервис (SendGrid, EmailJS, Firebase Extensions для отправки email).

### 🔴 HIGH: Отсутствие HTTPS
- **Severity**: HIGH
- **Описание**: Сайт работает на HTTP (localhost:3000), форма входа передаёт пароли в открытом виде.
- **Рекомендация**: При деплое на production обязательно настроить HTTPS (Let's Encrypt, Cloudflare и т.д.).

### 🟠 HIGH: Пароль хранится в SharedPreferences (Flutter-приложение)
```dart
users.add({'uid': 'admin', 'email': 'admin@numino.ru', 'password': 'admin123', ...});
```
- **Severity**: HIGH
- **Описание**: В Flutter-приложении используется локальное хранилище `SharedPreferences`, где пароли хранятся в **plaintext**.
- **Рекомендация**: Использовать `flutter_secure_storage` для хранения чувствительных данных.

### 🟠 HIGH: Driver видит Owner Dashboard (Flutter App — AuthGate)
```dart
if (auth.isOwner) return const OwnerDashboardScreen();
return const OwnerDashboardScreen(); // другие роли — позже
```
- **Severity**: HIGH
- **Описание**: Если входит водитель (role = 'driver'), вместо Dashboard водителя показывается Dashboard владельца парка. Это **нарушение ролевой модели** и потенциальная уязвимость RBAC (Role-Based Access Control).
- **Рекомендация**: Реализовать маршрутизацию в зависимости от роли.

### 🟠 HIGH: enforceAppCheck: false на всех Cloud Functions
- **Severity**: HIGH
- **Описание**: Во всех Firebase Cloud Functions отключён App Check. Это значит, что любой может вызывать эти функции, зная project ID.
- **Рекомендация**: Включить App Check (`enforceAppCheck: true`) на всех функциях.

### 🟠 MEDIUM: Сообщение об ошибке дублируется при каждом нажатии (main.js)
```javascript
msgEl.textContent = 'Неверный email или пароль.';
// ...
var hint = document.createElement('p');
hint.textContent = 'Демо: admin@numino.ru / admin123';
msgEl.appendChild(hint);  // <-- добавляется новый p при каждой ошибке!
```
- **Severity**: MEDIUM
- **Описание**: При каждом неверном вводе в модальное окно добавляется новый элемент `<p>` с подсказкой. Если нажать "Войти" несколько раз с неверным паролем, подсказки накопятся.
- **Рекомендация**: Заменить `appendChild` на замену текста, или удалять старую подсказку перед добавлением.

### 🟡 MEDIUM: duplicate `allow read` в Firestore Rules (trips)
```
allow read: if isAuth() && (resource.data.driverId == request.auth.uid || ...);  // строка 95
// ...
allow read: if isOwner() && ...;  // строка 115 — ДУБЛИКАТ, переопределяет первую
```
- **Severity**: MEDIUM
- **Описание**: Дублирующиеся правила чтения для коллекции `trips`. В Firestore Security Rules последний `allow read` переопределяет предыдущие, что может привести к непредсказуемому поведению.
- **Рекомендация**: Удалить дублирующееся правило.

### 🟡 MEDIUM: `addTrackPointsBatch` — неэффективная запись в цикле
```typescript
for (const pt of points) {
    batch.update(tripRef, { track: FieldValue.arrayUnion(newPoint), updatedAt: now });
}
```
- **Severity**: MEDIUM
- **Описание**: В батчевой записи каждая итерация делает `arrayUnion` с отдельной точкой. Это может вызвать превышение лимита операций в батче (500 операций), а также проблемы с конкурентностью.
- **Рекомендация**: Формировать массив точек вне батча и делать один update: `{ track: FieldValue.arrayUnion(allPoints) }`.

## 3. 🟡 ЗАМЕЧАНИЯ (Minor / Info)

### 🟡 Высокая нагрузка трека (Trip.track)
- **Описание**: GPS-точки пишутся раз в минуту. Для рейса длительностью 10 дней это ~14 400 точек. Firestore документ имеет лимит 1 MB.
- **Рекомендация**: Подумать о выносе трека в отдельную коллекцию или подколлекцию.

### 🟡 Путеводитель по карусели — авто-прокрутка не останавливается
- Карусель не приостанавливается при взаимодействии с кнопками навигации (кроме наведения мыши). Пользователь нажал кнопку, а через 5 секунд слайд переключился автоматически.

### 🟡 Отсутствует `noindex` для production
- Нет мета-тега `<meta name="robots" content="noindex">` для пароля или админ-панели. Хотя админка на отдельном маршруте, её можно проиндексировать.

### 🟡 Регистрация на лендинге — симуляция
- Регистрация в модальном окне создаёт учётную запись только в памяти браузера. При обновлении страницы данные теряются. Вводит пользователей в заблуждение.

### 🟡 "Войти" в footer ведёт напрямую в admin/index.html
- Ссылка `<a href="admin/index.html">Войти</a>` обходит модальное окно логина. Это может быть сделано намеренно, но нарушает единообразие UX.

### 🟡 Нет 404 страницы
- Если перейти на несуществующую страницу — сервер Python отдаёт стандартный листинг директории.

### 🟡 Нет accessibility-атрибутов
- Карусели не имеют `aria-live="polite"` для озвучивания смены слайдов.
- Отсутствуют skip-link для навигации с клавиатуры.
- Нет поддержки `prefers-reduced-motion`.

9. **Форма обратной связи имеет honeypot** — защита от ботов-спамеров.
