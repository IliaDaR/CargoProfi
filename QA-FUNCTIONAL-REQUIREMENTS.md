# Функциональные требования проекта CargoProfi (Numino)

## 🎯 Концепция
Сервис "Рабочий кабинет перевозчика" — SaaS-решение для управления автопарком. Два типа пользователей: **Владелец парка (Owner)** и **Водитель (Driver)**.

---

## 1. 👤 АВТОРИЗАЦИЯ И РОЛЕВАЯ МОДЕЛЬ

### 1.1. Регистрация и вход
- **Веб-лендинг (landing page)** — форма входа/регистрации в модальном окне
- **Flutter-приложение** — экраны входа и регистрации
- Учётные записи: `admin@numino.ru`, `owner@numino.ru` + пароль
- Роли пользователей: `owner` (владелец парка), `driver` (водитель)
- Сохранение сессии: в вебе — localStorage (заглушка), в Flutter — SharedPreferences

### 1.2. Ролевой доступ
- **Владелец:** видит всё — все машины, все рейсы, все расходы, зарплату всех водителей
- **Водитель:** видит только свои рейсы и свои расходы
- Права проверяются: Firestore Security Rules (сервер) + логика на клиенте

---

## 2. 🚛 УПРАВЛЕНИЕ АВТОПАРКОМ (Owner)

### 2.1. Автомобили (Vehicles)
- **Добавление машины:** госномер, марка, модель, год, VIN, тип топлива
- **Список всех машин** с индикацией статуса: "в рейсе" / "свободен"
- **История рейсов** по каждому автомобилю
- **Статистика использования:** общий пробег, количество рейсов

### 2.2. Водители (Drivers)
- **Добавление водителя:** ФИО, email, телефон, номер прав
- **Назначение** водителя на автомобиль
- **Список водителей** с привязкой к владельцу

### 2.3. Дашборд (Dashboard)
- Обзор парка: количество машин, в рейсе / свободны
- Количество завершённых / активных рейсов
- Общий доход по всем рейсам
- Карточки со статистикой (виджеты)

---

## 3. 🗺️ GPS-ТРЕКИНГ РЕЙСОВ (Driver + Owner)

### 3.1. Начало рейса (Driver)
- Водитель нажимает "Начать рейс"
- Фиксируется: дата/время старта, GPS-координаты (широта, долгота)
- Указывается: маршрут, описание груза
- Создаётся документ рейса в Firestore со статусом `active`
- Проверка: нельзя начать новый рейс, если есть активный

### 3.2. GPS-трекинг (автоматический)
- Каждые **60 секунд** фиксируются координаты
- При обрыве связи — **офлайн-буфер**: точки копятся локально и синхронизируются при подключении
- Пробег считается по **формуле гаверсинусов** (haversine) — сумма расстояний между всеми GPS-точками
- Батчевая отправка точек для экономии трафика (`addTrackPointsBatch`)

### 3.3. Завершение рейса (Driver)
- Фиксация времени возврата и конечных GPS-координат
- **Авторасчёт пробега:** если трек не пуст
- **Ручной ввод пробега:** если GPS-данных недостаточно
- Указывается доход за рейс
- Статус меняется на `completed`

### 3.4. Просмотр рейсов
- **Driver:** история своих рейсов (статус, даты, маршрут, пробег, доход)
- **Owner:** все рейсы всех водителей с фильтрацией по статусу и периоду

---

## 4. 💰 УЧЁТ РАСХОДОВ (Driver + Owner)

### 4.1. Добавление расхода (Driver)
- Во время **активного рейса** водитель может добавить расход:
  - **Сумма** (обязательно, положительное число)
  - **Категория** (из 8 предустановленных):
    - `fuel` — топливо
    - `parking` — стоянка
    - `repair` — ремонт
    - `toll` — дорожный сбор
    - `washing` — мойка
    - `tires` — шины
    - `insurance` — страховка
    - `other` — прочее
  - **Описание** (необязательно)
  - **Фото чека** (загружается в Firebase Storage, до 10 MB, только изображения)
  - **GPS-координаты** и **время** прикрепляются автоматически

### 4.2. Просмотр расходов (Owner)
- Сводная таблица расходов по категориям
- Сводка по расходам водителя за период
- Возможность открыть фото чека
- Суммирование расходов по категориям (отчёт `byCategory`)

---

## 5. 📄 ПУТЕВЫЕ ЛИСТЫ (Owner)

### 5.1. Генерация PDF
- Генерация по форме, утверждённой **Приказом Минтранса России**
- Доступно только **для завершённых рейсов**
- **Автоматическое заполнение полей:**
  - Дата (startTime)
  - Госномер автомобиля (plateNumber)
  - Марка, модель (vehicle.brand, vehicle.model)
  - ФИО водителя (driver.displayName)
  - Номер водительского удостоверения (driver.licenseNumber)
  - Время выезда (startTime)
  - Время возврата (endTime)
  - Пробег (mileage)
  - Маршрут (routeDescription)
  - Груз (cargoDescription)
  - Доход за рейс (income)

### 5.2. Хранение
- PDF сохраняется в **Firebase Storage**: `/waybills/{ownerId}/{tripId}.pdf`
- Ссылка на PDF записывается в документ рейса (`waybillUrl`)

---

## 6. 💵 РАСЧЁТ ЗАРПЛАТЫ (Owner)

### 6.1. Правила начисления
- Владелец задаёт правило для каждого водителя:
  - **Процент от дохода рейса** (`percent`): 15%, 20%, и т.д.
  - **Фиксированная сумма за рейс** (`fixed`): 5 000 ₽, 10 000 ₽, и т.д.
- Правила могут быть обновлены (старые деактивируются)
- Активно только **одно правило** на водителя

### 6.2. Расчёт зарплаты
- Выбор периода (startDate — endDate)
- Сбор всех завершённых рейсов водителя за период
- Расчёт:
  - `percent`: зарплата = сумма(доход рейсов) × percentValue / 100
  - `fixed`: зарплата = количество рейсов × fixedValue
- Сохранение результата в `salaryPayments` со статусом `calculated`
- История расчётов: владелец видит все, водитель — только свои

## 8. 📱 Flutter-приложение (Mobile)

### 8.1. Экраны
1. **Экран входа / регистрации** (LoginScreen / RegisterScreen)
2. **Дашборд владельца** (OwnerDashboardScreen) — статистика парка

4. **Рейсы** (TripsScreen) — просмотр рейсов

## 9. 🛡️ ФУНКЦИОНАЛ БЕЗОПАСНОСТИ

### 9.1. Firestore Security Rules
- Разграничение прав по ролям (owner/driver)
- Водитель читает/пишет только свои данные
- Владелец управляет всеми данными своего парка
- Запрет удаления расходов и salaryPayments
- Только водитель может создавать/обновлять рейсы
- Только владелец может управлять машинами, правилами зарплаты

### 9.2. Firebase Storage Rules
- Чеки: водитель → только в свою папку, только изображения, до 10 MB
- Путевые листы: только владелец может создавать и читать
- Запрет на удаление и перезапись

### 9.3. Cloud Functions
- Аутентификация через `request.auth?.uid` на всех функциях
- Валидация входных данных (обязательные поля, типы)
- Проверка принадлежности: водитель управляет только своими рейсами
- Ограничение на размер и формат данных
