# Gap Analysis: Требования vs Реальная реализация

## 📊 Легенда

| Статус | Значение |
|---|---|
| ✅ **Реализовано** | Функция полностью работает |
| ⚠️ **Частично** | Функция есть, но с ограничениями/багами |
| ❌ **Не реализовано** | Функция отсутствует или не работает |
| 🔧 **Только backend** | Cloud Function есть, но нет UI-вызова |

## 1. 🚛 УПРАВЛЕНИЕ АВТОПАРКОМ

### Требование: Добавлять машины (госномер, марка, модель, VIN)

| Компонент | Статус | Детали |
|---|---|---|
| **Flutter App (экран)** | ✅ | `vehicles_screen.dart` — диалог добавления авто (госномер, марка, модель) |
| **Cloud Function** | ❌ | Нет API-функции для добавления машины — только прямой доступ через Firestore |
| **Firestore Rules** | ⚠️ | Есть `allow create` для vehicles, но VIN не обязателен в коде |

**Проблема:** При добавлении авто в UI собираются только: госномер, марка, модель. **VIN, год, тип топлива** — не запрашиваются, хотя есть в модели Vehicle.

**Gap:** `vin`, `year`, `fuelType` отсутствуют в форме добавления.

### Требование: Добавлять водителей (ФИО, телефон, права)

| Компонент | Статус | Детали |
|---|---|---|
| **Flutter App (экран)** | ⚠️ | `vehicles_screen.dart` — диалог добавления (только ФИО + телефон) |
| **Cloud Function** | ❌ | Нет API для создания водителей |
| **Firestore Rules** | ⚠️ | Есть, но email, licenseNumber не собираются |

**Проблема:** При добавлении водителя собираются только `ФИО` и `Телефон`. **Email** и **Номер прав** — не запрашиваются, хотя есть в модели DriverProfile.

**Gap:** `email`, `licenseNumber` отсутствуют в форме добавления водителя.

### Требование: Назначать водителей на машины

| Компонент | Статус | Детали |
|---|---|---|
| **Flutter App** | ❌ | Нет UI для назначения водителя на машину |
| **Модель Vehicle** | ✅ | Есть поле `activeDriverId` |
| **Данные** | ⚠️ | В демо-данных `v1` привязан к `d1`, `v3` к `d2` (хардкод) |

**Gap:** Владелец не может через интерфейс назначить водителя на конкретную машину. Статус "в рейсе" определяется наличием активного рейса в Firestore, а не прямым назначением.

## 3. 🗺️ GPS-ТРЕКИНГ РЕЙСОВ В РЕАЛЬНОМ ВРЕМЕНИ

### Требование: GPS-трекинг каждые 60 секунд, формула гаверсинусов

| Компонент | Статус | Детали |
|---|---|---|
| **LocationService** | ✅ | `location_service.dart.bak` — запрос разрешений, периодический опрос GPS, стрим позиций |
| **TripProvider** | ✅ | `trip_provider.dart` — запуск трекинга раз в 60 сек, локальный трек, расчёт пробега по гаверсинусам |
| **Cloud Function startTrip** | ✅ | `trips.ts` — создание рейса с первой GPS-точкой |
| **Cloud Function addTrackPoint** | ✅ | `trips.ts` — добавление точки к треку |
| **Cloud Function addTrackPointsBatch** | ⚠️ | Есть, но с багом: каждая точка в отдельном `batch.update` + `arrayUnion` |
| **Cloud Function endTrip** | ✅ | `trips.ts` — завершение с авторасчётом пробега |
| **Формула гаверсинусов** | ✅ | Реализована в `distance.ts` (backend) и `distance.dart` (Flutter) |
| **Офлайн-буфер** | ⚠️ | В `local_storage.dart` нет явной реализации офлайн-буфера для GPS-точек. Только локальный трек в памяти `TripProvider._localTrack` |

**Gap #1:** `addTrackPointsBatch` — неэффективная запись. Каждая точка добавляется отдельным `arrayUnion` в цикле. Правильно: собрать все точки в массив и сделать один update.

**Gap #2:** Офлайн-буфер заявлен на лендинге ("При обрыве связи точки копятся локально и синхронизируются при подключении"), но в коде не реализован. Точки хранятся только в памяти (`_localTrack`), не сохраняются на диск для последующей синхронизации.

| Компонент | Статус | Детали |
|---|---|---|
| **Cloud Function generateWaybill** | ✅ | `pdf.ts` — генерация PDF через pdfkit, все поля рейса |
| **Firebase Storage** | ✅ | PDF сохраняется в `/waybills/` и делается публичным |

| **Flutter App — просмотр PDF** | ❌ | Нет кнопки для открытия ссылки waybillUrl |

**Gap (CRITICAL):** Cloud Function `generateWaybill` написана, но **нигде не вызывается из UI**:
- В `trips_screen.dart` (owner) нет кнопки "Сформировать путевой лист"

**Итог:** Путевые листы — **мёртвая функция**. Код есть, но пользователь не может сгенерировать PDF через интерфейс.

## 6. 💵 РАСЧЁТ ЗАРПЛАТЫ

### Требование: Процент от дохода или фиксированная сумма за рейс, с историей

| Компонент | Статус | Детали |
|---|---|---|
| **Flutter App (Owner)** | ⚠️ | `salary_screen.dart` — расчёт есть, но **локальный** (без Cloud Functions) |
| **Cloud Function calculateSalary** | ✅ | `salary.ts` — серверный расчёт, валидация, сохранение |
| **Cloud Function setSalaryRule** | ✅ | `salaryRules.ts` — управление правилами |
| **Cloud Function getSalaryHistory** | ✅ | `salary.ts` — история расчётов |
| **Firestore Rules** | ✅ | Проверка принадлежности owner/driver |
| **SalaryProvider** | ✅ | Обёртка над Cloud Functions |

**Gap #1 (CRITICAL):** `salary_screen.dart` **игнорирует Cloud Functions** и делает расчёт **локально** через `LocalStorage`:
```dart
store.addSalaryPayment(SalaryPayment(...)); // локально, не вызывает calculateSalary
```

**Gap #2:** Не используются `setSalaryRule` и `SalaryProvider`. Пользователь просто вводит процент/сумму вручную каждый раз. Нет сохранения "правила" для водителя.

**Gap #3:** История зарплаты грузится из локального `store.salaryPayments`, а не из Firebase через `getSalaryHistory`.

## 7. 🔄 ПРОЧИЕ ПРОБЛЕМЫ

### 7.1. Firestore vs LocalStorage — конфликт архитектуры

В Flutter-приложении **два параллельных слоя данных**:

| Компонент | Хранилище | Статус |
|---|---|---|
| **Owner Dashboard** | `LocalStorage` (SharedPreferences) | ✅ Данные из демо |
| **Vehicles** | `LocalStorage` | ✅ |
| **Trips (Driver)** | `Firestore` (через Cloud Functions) | ✅ |
| **Expenses (Driver)** | `Firestore` (через Cloud Functions) | ✅ |
| **Trips (Owner)** | `LocalStorage` | ⚠️ Owner видит только демо-рейсы |
| **Expenses (Owner)** | `LocalStorage` | ⚠️ Owner видит только демо-расходы |
| **Salary** | `LocalStorage` | ⚠️ Локально, не через Cloud |

**Проблема:** Водитель использует Firebase, а владелец использует локальные демо-данные. Это значит, что владелец **не видит рейсы, созданные водителем через Firebase**.

### 7.2. Flutter Web Admin не подключён к Firebase

`website/admin/index.html` — это собранный Flutter Web App. Но в коде `main.dart` используется `LocalStorage`, а не Firebase. При входе через лендинг (admin@numino.ru) пользователь попадает на Flutter Web App, который использует локальное хранилище.

### 7.3. .bak файлы — не используются

`firestore_service.dart.bak`, `location_service.dart.bak`, `auth_service.dart.bak` — это `*.bak` файлы. Они **не компилируются** в Flutter при сборке. 

TripProvider и ExpenseProvider импортируют:
```dart
import '../services/firestore_service.dart';  // без .bak
import '../services/location_service.dart';    // без .bak
```

Но таких файлов без `.bak` в папке `services` нет (см. `list_files` — только файлы с `.bak`).

**Итог:** **Flutter-приложение не скомпилируется** — файлы сервисов имеют расширение `.bak`, а импорты в коде ссылаются на файлы без `.bak`.
